## Context

`docling-ingestion-adapter` emits `source_document.json` but ships no shared validator; each downstream change (3–9) cites `source-document/v1` anchors and needs the shape frozen and testable first (`ROADMAP.md` Phase 2, "Maturity and dependency policy").
The repo's verification command is `python3 -m unittest discover -s tests`; fixtures must be deterministic and repo-owned (`ROADMAP.md` verification plan). The facts-only constraint is defined in `CONTEXT.md` ("Source Document") and ADR boundaries.

## Goals / Non-Goals

**Goals:**

- A single reusable validation entry point for `source-document/v1` callable by the adapter and downstream changes.
- Deterministic repo-owned fixtures that exercise pages, blocks, bbox, table cells, captions, headers, footers, and notes.
- Structured, field-path-naming errors so callers can act on failures.

**Non-Goals:**

- Adding or changing `source-document/v1` fields (owned by change 1).
- Any exhibit interpretation, case-evidence migration, or export formatting.
- Mandatory image fixtures or real EMBA PDFs.

## Decisions

### Pure-stdlib validator over a schema framework dependency

Implement validation as plain Python over the parsed dict rather than adding a JSON-Schema library.
Rationale: the repo avoids introducing dependencies unless requested; the contract is small and field-path error reporting is easier to control by hand. Verify behavior with `unittest`.
Alternative considered: `jsonschema` — rejected to avoid a new dependency and to keep facts-only semantic checks (which JSON Schema expresses awkwardly) in one place.

### Facts-only enforcement via a reserved interpretation-key denylist

Detect interpretation leakage by rejecting reserved keys that belong to `exhibit_semantics.json` (e.g. exhibit type, subtotal meaning, row role) appearing inside a source document.
Rationale: source-document/v1 is positively specified for facts; a denylist of interpretation keys is a tractable guard that matches the `CONTEXT.md` glossary split.
Alternative considered: full semantic analysis — rejected as over-engineering for a structural validator.

### Fixtures generated by repo-owned code, not committed binaries where avoidable

Prefer generating the PDF fixture from a small script so the fixture is deterministic and reviewable; treat any committed binary as a repo-owned artifact under the size budget.
Alternative considered: commit real case pages — rejected by the verification plan (real PDFs are exploratory only).

## Implementation Contract

- **Behavior**: a validation function (e.g. `validate_source_document(doc) -> Result`) returns success for a conforming `source-document/v1` document and a structured failure naming the first offending field path otherwise; it rejects documents carrying interpretation keys. An optional thin CLI wrapper validates a file path and exits non-zero on failure.
- **Interface / data shape**: input is a parsed `source_document.json` dict (or path for the CLI); output is a result object exposing `ok: bool` and `errors: list` where each error names a field path such as `pages[0].height` or `tables[2].cells`. The reserved interpretation-key set is part of this contract.
- **Failure modes**: missing required field → failure naming the path; interpretation key present → failure naming the disallowed key; malformed JSON (CLI) → non-zero with a parse error. No failure is silent.
- **Acceptance criteria**: the spec scenarios in `specs/source-document-validation/spec.md` hold against the repo-owned fixtures; `python3 -m unittest discover -s tests` stays green; the generated PDF fixture exercises pages, a table with cells, a caption, a header, and a footer.
- **Scope boundaries**: in scope — validator module, fixtures, tests, optional CLI wrapper. Out of scope — changing the v1 field set, semantic interpretation, exporter/`case_spec.json` work.

## Risks / Trade-offs

- [Denylist misses a novel interpretation key] → keep the reserved set centralized and extend it as `exhibit_semantics.json` keys are defined in change 3; structural checks still pass independently.
- [Hand-rolled validator diverges from adapter output] → the adapter's normalization and this validator share the `source-document/v1` field list as the single source of truth; a contract-drift test ingests the generated fixture through both.

## Migration Plan

Additive — new validation module, fixtures, and tests. No change to existing artifacts. Rollback is deletion of the new files.

## Open Questions

- The exact reserved interpretation-key set is finalized jointly with `exhibit-semantics-foundation` (change 3); the structural-validation portion is independent and does not block on it.
